- **Date:** 2026-07-07
- **Status:** approved design — ready for implementation planning
- **Depends on:** M2 MVP CLI (shipped; see `docs/specs/2026-07-07-m2-mvp-cli-design.md`), M1 anatomy contract

## Decisions (adjudicated 2026-07-07)

| Decision | Choice | Rejected |
|---|---|---|
| Packaging | Skill source in repo at `skills/using-shakespii/`; installed via symlink into `~/.claude/skills/using-shakespii` | Repo-only without install (untriggerable, no real dogfood); copy into live dir (two copies drift) |
| Granularity | One skill, Procedure branches into audit loop and authoring loop | Two skills (duplicates ~80% of content — the XS01 clone smell shakespii itself will flag at M3) |
| Dogfood gate | Zero findings — no errors *and* no warnings; repo test asserts exit 0 and `findings: []` | Exit-0-only (weakens the flagship claim); pinned-warnings snapshot (bookkeeping without signal) |
| Content architecture | Thin SKILL.md plus `references/rule-remediations.md` sibling with per-rule fix recipes | All-inline (bloats at M3, dogfoods less linter surface); generated reference (needs rule-metadata code — M3+ material) |

## 1. Scope and rationale

Humans instruct agents; agents drive the CLI. The shakespii CLI is the deterministic
substrate (exit codes, rule IDs, `--json`, zero LLM calls); `using-shakespii` is the thin
operational interface that teaches an agent two workflows:

- **Audit loop:** lint an existing skill, interpret findings by rule ID, fix, re-lint.
- **Authoring loop:** scaffold a new skill with `init`, fill the anatomy sections,
  lint-loop until clean, fill evals, present to the human.

The skill is recursive dogfood: it is authored through its own authoring loop (starting
from literal `shakespii init using-shakespii` output), it must itself lint to zero
findings, and it ships its own evals. It is the first real-world UX test of the M1
scaffold template.

M2.5 is a content milestone. It adds no CLI features. Any CLI change is in scope only if
dogfood friction *forces* it, and such changes are triaged explicitly, never folded in
silently.

## 2. Skill shape and placement

### Layout

```
skills/using-shakespii/
├── SKILL.md                        # thin operational layer, 7 anatomy sections
├── README.md                       # human-facing, from init scaffold
├── references/
│   └── rule-remediations.md        # per-rule fix recipes, provenance-pinned
└── evals/
    └── evals.json                  # 4 real cases, skill-creator schema
```

`references/rule-remediations.md` is not part of the scaffold; it is added during
authoring. The SKILL.md link to it must resolve through the sibling inventory (ST02).

### Install

```sh
ln -s <repo>/skills/using-shakespii ~/.claude/skills/using-shakespii
```

Documented in the repo README with an uninstall note (`rm` the link). Creating the
symlink is an execution-time step; it adds one new entry to the live skills directory and
touches nothing that already exists there.

### Frontmatter

```yaml
---
name: using-shakespii
description: "Use when creating a new agent skill or auditing, linting, or fixing an existing one — drives the shakespii CLI (init, lint --json) to scaffold standard SKILL.md skills and resolve findings until clean."
version: 0.1.0
---
```

- `name` equals the directory name, kebab-case (FM02).
- The description starts with a profile trigger prefix (`use when` — FM04), is third
  person, and carries concrete searchable keywords (skill, lint, scaffold, SKILL.md).
  Exact final wording may be tuned during authoring, but it must keep the trigger-first
  prefix and stay under 500 characters (FM03 warn threshold).
- `version` is present per the profile's FM05 (unimplemented at M2; template ships it).

### Body

Exactly the seven anatomy sections with canonical headings — Intent, Inputs,
Preconditions, Procedure, Output, Examples, Anti-patterns. Target well under the ST01
budget (~150 lines): prose for judgment steps, exact commands for fragile ones.

Although only the six seed rules are enforced at M2, the skill is written to clear the
*full* profile by intent — description under 500 chars (FM03), all seven sections present
(CT04/CT05/CT06), size budget (ST01), forward-slash paths (HY01), no absolute paths
(HY02) — so the zero-findings gate keeps holding as M3 rules land.

### Preconditions content

The skill's Preconditions section enumerates: Bun installed; shakespii cloned and linked
(`bun install && bun link` in the repo, `shakespii --version` resolves); target skill
directory readable (audit) or parent directory writable (authoring).

## 3. Procedure: the two loops

### Shared core (both branches)

1. Run `shakespii lint <dir> --json`; parse stdout — `version: 1` schema; each finding
   carries `{ruleId, severity, file, line, message}`.
2. Exit-code discipline: `0` = no errors, proceed; `1` = errors, enter the fix loop;
   `2` = lint itself could not run — report the stderr message verbatim and stop. Never
   guess around a broken run.
3. Per finding: look up `ruleId` in `references/rule-remediations.md`, apply the
   *minimal* fix, re-lint.
4. Loop until exit 0, then handle warnings: fix them, or surface each unfixed warning to
   the human explicitly. Never silently ignore a warning.
5. Forward compatibility: a finding whose `ruleId` has no reference entry (new rules land
   before the reference catches up) is fixed from the finding `message` itself — messages
   are written to be actionable. The skill never stalls on an unknown rule.

### Audit branch — lint/fix an existing skill

1. Identify the target skill directory (the user names it). Multiple skills = iterate
   directories, one lint each. `--corpus` mode is future CLI (M3); the skill must not
   reference it.
2. Lint, fix per the remediation reference, re-lint.
3. Fixes preserve the skill's voice and intent: reword a description to be trigger-first,
   do not rewrite the skill's purpose. Content-weakening to appease the linter is a named
   anti-pattern (mirror of "never weaken tests").
4. Report: before/after finding counts, what changed per rule, remaining warnings with
   reasons.

### Authoring branch — create a new skill

1. Before scaffolding, confirm with the human: skill name (kebab-case), purpose, trigger
   situations. A wrong name means FM02 findings plus directory-rename churn.
2. `shakespii init <name>` in the agreed parent directory.
3. Fill all seven sections; every `TODO(shakespii):` token must go (PH01 fires per
   occurrence, across SKILL.md, README.md, and evals.json).
4. Examples section: at least one real input→output worked example with concrete values
   (CT03 checks the ordered marker pair; a trigger-phrase list alone fails).
5. Fill `evals/evals.json`: at least 3 cases, one a near-miss negative.
6. Lint-loop per the shared core.
7. **Present, don't install.** Hand the human the skill content, the clean lint output,
   and the evals. Installing into `~/.claude/skills/` requires explicit human approval,
   and never happens with findings outstanding.

## 4. `references/rule-remediations.md`

Header block pins provenance (HY04 discipline): profile `default`, M2 vintage,
last-reviewed date, and a note that entries track `profiles/default.yaml`.

One entry per seed rule, uniform shape — **Contract / Common cause / Fix /
Before→After snippet** (snippets 2–4 lines each; ~80–100 lines total):

| Rule | Fix recipe teaches |
|---|---|
| FM01 | Repair YAML; add missing `name`/`description`; remove or justify unknown fields |
| FM02 | Kebab-case, ≤64 chars, must equal the directory name — rename the field or the directory deliberately, never both blindly |
| FM04 | Rewrite the description trigger-first ("Use when…", third person); before/after shows "I help you write skills" → "Use when creating…" |
| CT03 | Replace trigger-phrase lists with a real worked example — input then output, concrete values; a leftover scaffold token in Examples also fires this |
| ST02 | Link target must exist inside the skill directory — create the file, fix the path, or drop the link; `../` is never allowed |
| PH01 | Every `TODO(shakespii):` occurrence is a finding — write real content; deleting the section is not a fix |

M3 extends this file by appending entries; SKILL.md is untouched by catalog growth.

## 5. Evals

`evals/evals.json` in the skill-creator schema (the shape the init stub carries),
data-only until the M4 harness executes them. Four cases:

1. **`using-shakespii-audit-existing`** (positive) — prompt: "Lint my skill at
   `~/.claude/skills/caveman` and fix what it finds." Expectations: invokes
   `shakespii lint <dir> --json` rather than eyeballing files; consults the remediation
   reference per ruleId; re-lints after fixing; reports before/after counts.
2. **`using-shakespii-author-new`** (positive) — prompt: "Create a new skill that teaches
   agents to review Dockerfiles." Expectations: confirms name/purpose/triggers before
   scaffolding; uses `shakespii init` (never hand-rolls the layout); zero
   `TODO(shakespii):` tokens remain; presents with clean lint output and asks approval
   before any install.
3. **`using-shakespii-near-miss-code-lint`** (negative) — prompt: "Fix the ESLint errors
   in src/." Expectation: the skill does **not** trigger — shakespii lints agent skills,
   not source code. This is the TR02-style near-miss the description's keywords must not
   swallow.
4. **`using-shakespii-lint-run-failure`** (error path) — prompt: lint a directory with no
   SKILL.md. Expectations: recognizes exit 2, reports the failure verbatim, does not loop
   or fabricate findings.

`expected_output` per case: one or two sentences describing good agent behavior;
`expectations`: the checkable assertions above. TR01 (minimum 3 cases) cleared with
margin.

## 6. Repo integration and testing

### The weld test

`tests/skill/using-shakespii.test.ts` spawns the real CLI (same mechanism as
`tests/cli/keystone.test.ts`) against the checked-in skill:

- `shakespii lint skills/using-shakespii --json` → exit 0 **and** `findings: []`
  (the zero-findings gate — errors and warnings both empty).
- `evals/evals.json` parses as JSON and contains ≥ 4 cases (the linter does not
  schema-check evals until M4; the test covers that gap).

Note the two bars are deliberately different: the zero-findings gate is the companion
skill's *own* standard (Decisions table), stricter than the exit-0-plus-warnings-handled
bar the skill teaches for arbitrary target skills in §3.

### TDD shape

The milestone's TDD cycle *is* the authoring loop. RED is observed, not committed: run
the weld-test assertions against the raw scaffold first — it fails with exactly the
keystone finding set (18 PH01 + 1 FM04 + 1 CT03) plus the evals-count assertion — then
authoring drives it GREEN. Every commit keeps the suite green:

1. First commit: raw `shakespii init using-shakespii` output only, byte-identical to the
   template modulo `{{name}}` (no test yet, suite stays green). Git history proves the
   skill started life as its own scaffold.
2. Subsequent commits: fill sections and add the weld test together once GREEN, following
   the M2 precedent of committing test + implementation after RED is observed and
   recorded in the plan's verification log.

### Docs and close-out

- ROADMAP: tick both M2.5 boxes.
- Repo README: status line update plus an install section (symlink command + uninstall).
- Spec dual-location sync (canonical `~/.ai-pref-nsync/local-docs/ai-shakespii/specs/`,
  repo `docs/specs/` mirror), as with M1/M2.
- Template friction found while authoring through the scaffold: recorded in
  `docs/DOGFOOD-M2.5.md` (created only if non-empty) — feeds M3 template and rule tuning.

## 7. Out of scope

- `--corpus` mode and the remaining 22 catalog rules (M3).
- Eval runner / `shakespii test` (M4).
- Writer-as-skill full authoring loop (M5).
- Personal-skill migration (open decision — untouched).
- CLI code changes, except those forced by dogfood friction and explicitly triaged.

- [ ] Weld test green: `lint skills/using-shakespii --json` → exit 0, `findings: []`;

- [ ] Skill installed via symlink at `~/.claude/skills/using-shakespii` and triggerable.
- [ ] Full test suite and `tsc --noEmit` green.
- [ ] ROADMAP and README updated; spec synced to both doc locations.
